✅ 3 Good Patterns

Dataclass for Clear Structure

The Cube dataclass provides a strongly-typed, immutable data container with field validation and automatic __init__, __repr__, and __eq__ methods.

Basics: Using @dataclass removes boilerplate; each field has a declared type (x: float, size: int, etc.).
Engineering: Type safety reduces bugs when passing Cube instances between functions. Immutable-friendly design supports functional programming patterns.
Architecture: Dataclass acts as a contract: any function receiving a Cube knows exactly what fields it has. Easier to refactor or extend (e.g., add acceleration field).

Separation of Concerns

Each function has a single responsibility: find_bigger_neighbors filters cubes, compute_flee_steering calculates direction, apply_steering modifies velocity. This makes logic testable and reusable.

Basics: find_bigger_neighbors() only filters; compute_flee_steering() only computes; apply_steering() only applies. No mixing of concerns.
Engineering: Each function is small and decoupled. You can test compute_flee_steering() without rendering or physics. Easy to swap algorithms (e.g., try different steering weights).
Architecture: The update_cube() function becomes a clear pipeline: threats → steering → velocity update → position update → collision. Highly maintainable.

Configuration Constants at Top

Behavior parameters (FLEE_RADIUS, FLEE_STEER, CUBE_COUNT, FPS) are declared as module-level constants, making tuning trivial without touching function logic.

Basics: Constants like FLEE_RADIUS = 140 and FLEE_STEER = 1200 are centralized. Changing one value adjusts the whole system's behavior.
Engineering: Non-developers can tweak parameters without understanding code. Supports rapid prototyping and experimentation (e.g., "What if FLEE_STEER = 2000?").
Architecture: An ideal pattern for game development and simulations. Future readers know where to look for "magic numbers." Could evolve into config files or UI sliders.

⚠️ 2 Potential Issues

🔴 O(n²) Distance Calculations

In find_bigger_neighbors(), every frame iterates all cubes for every cube, computing distances. With 30 cubes, that's ~900 distance calculations per frame. At 60 FPS, unscalable.

Basics: In find_bigger_neighbors(), a nested loop over cubes with sqrt() distance calculation creates quadratic complexity.
Engineering: Performance degrades rapidly: 30 cubes is manageable; 100 cubes would lag. sqrt() is costly; consider squared distances for comparison or spatial partitioning (grid, quadtree).
Architecture: If this simulation is meant to scale to thousands of entities, a spatial indexing data structure (grid or quadtree) is essential. For now, document the limit and consider a TODO.

🔴 Dead Code & Inconsistency

The in_radius() function duplicates distance logic but is never called. Also, velocity is randomly scaled (0.95–1.05) on bounce, causing subtle speed drift over time.

Basics: in_radius() calculates distance but isn't imported or called anywhere. The random bounce factor might be unintended.
Engineering: Dead code accumulates technical debt. The random bounce factor is poorly documented—is it intentional for realism, or a bug? Unclear intent slows debugging.
Architecture: Either remove in_radius() or refactor to use it (DRY principle). Document the random bounce with a comment explaining the game design rationale.

Performance Analysis

Score: 3 / 6 — Simulation-friendly but not optimized for scale. Main bottleneck is O(n²) neighbor distance calculations. Rendering and collision handling are efficient.

🚀 Performance Wins

Efficient Rendering

draw_cube() uses pygame.draw.circle() directly without expensive transformations. No per-pixel operations or shadow rendering. Blits to screen are fast.

Reasonable Frame Budget

With 30 cubes at 60 FPS, each frame has ~16ms. Current logic (neighbor search, steering, position update) fits comfortably. Only issue at higher cube counts.

🐢 Performance Risks

Quadratic Neighbor Search

find_bigger_neighbors() iterates all cubes for each cube (O(n²)). At 30 cubes, that's manageable; at 100+, this alone will dominate frame time. sqrt() per pair adds cost.

Repeated Distance Calculations

Center points (cx, cy) and (nx, ny) are recalculated in find_bigger_neighbors() and compute_flee_steering(). Caching them per frame would halve distance math overhead.

Score: 3 / 6 — Acceptable for 30-50 cubes; upgrade spatial partitioning for 100+.
